## 摘要

- 这个任务评估 GitSonar 是否应该从当前内嵌 HTML/CSS/JS 迁移到更现代的前端构建体系。
- 现在做评估是因为 P0/P1 的产品和 API 边界已经明显增强，但仍不能直接进行大规模前端重写。
- 结论：当前不启动 React / Vite 重写；继续沿用现有 Vanilla JS 分片与 JSON API，未来只在边界更稳定后做小步迁移实验。

## 评估结论

### 现在不应启动完整前端迁移

当前最稳妥的判断是：**不启动 React / Vite / TypeScript 重写**。原因如下：

- GitSonar 仍是 Windows 本地桌面工具，现有 Python + 本地 HTTP + 内嵌资源链路部署摩擦低。
- P0/P1 刚完成 API、事件、AI artifact 和聚类边界，小步稳定比换栈更重要。
- 当前测试已经围绕聚合 JS / CSS 合约建立保护，直接引入构建链会扩大验证面。
- 现代化本身不会自动解决状态模型、局部刷新、可访问性和交互复杂度问题。

### 当前阶段推荐做法

短期继续增强现有前端边界：

- 继续保持 `runtime_ui/js/*.py` 和 `runtime_ui/css/*.py` 分片，而不是回退到单文件。
- 对新增交互优先补 `test_runtime_ui_assets.py`、`test_runtime_ui_js_contract.py` 和 `test_ui_js_smoke.py`。
- 把业务数据继续推向 JSON API，减少模板一次性注入的长期负担。
- 对复杂前端功能优先提取纯函数或小型渲染 helper，便于字符串合约测试。

### 未来迁移前置条件

只有满足以下条件后，才建议重新打开前端现代化实现任务：

1. 静态壳和 JSON API 边界稳定，主数据不再依赖整页重建。
2. 至少有一组浏览器级 smoke test 或截图验证，能覆盖设置、发现、详情、更新和批量操作。
3. 明确打包产物如何包含前端构建输出，并能在无 Node 环境的用户机器上运行。
4. 迁移策略限定为局部试点，例如先迁移独立设置页或诊断页，而不是主工作台。
5. 回滚路径明确：可以在同一个 release 中切回现有 `runtime_ui` 聚合资源。

## 推荐后续路线

1. 保持当前实现，继续用 Vanilla JS 分片支撑 P2 剩余维护。
2. 新增复杂 UI 前，先补更细的 JS 合约测试。
3. 如果未来要引入构建链，先写独立计划并选择低风险页面试点。
4. 只有试点证明可维护性、测试和打包收益明显，再评估主工作台迁移。
