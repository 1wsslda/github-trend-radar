# 0007 - 发现结果“为什么推荐”解释

## 任务元信息

- 任务 ID：`GS-P0-007`
- 优先级：`P0`
- 当前状态：`[x]`

## 摘要

- 问题：发现结果的排序缺少可解释性，用户难以信任“为什么它排在前面”。
- 为什么现在做：高价值且改动相对可控，有助于提高发现系统的可信度。
- 结果：用户可以看到每个结果的推荐依据。

## 战略映射

- 战略：继续使用现有发现评分模型，只暴露对用户可理解的最小解释层。
- 路线图：对应 `docs/roadmap/ROADMAP.md` 中提升发现可信度的 P0 任务。
- Sprint：对应 `docs/sprints/CURRENT_TOP10.md` 候选排名 `7`。

## 当前状态

- 发现评分结果已统一产出 `match_reasons`。
- 发现页 Top 卡片、仓库卡片和详情面板都会展示“为什么推荐”。
- 推荐理由会按用户更容易理解的顺序输出，本地忽略反馈优先级高于热度 / 趋势等次级说明，避免被前 4 条理由截断。
- 未暴露内部权重细节，也不要求用户理解完整评分模型。

## 目标

- 暴露最小可解释信息。
- 不要求用户理解完整打分模型也能看懂主要原因。
- 保持现有排序逻辑可逐步演进。

## 非目标

- 不做复杂可视化评分系统。
- 不引入新的推荐模型。
- 不重写发现引擎。

## 范围

### 范围内

- 推荐原因字段
- 卡片或详情里的原因展示
- 基础文案整理

### 范围外

- 复杂权重编辑器
- 个性化推荐引擎
- 全量排序重构

## 架构触点

- 发现逻辑：`runtime_github/discovery.py`
- HTTP：`runtime/http.py`
- UI：`runtime_ui/js/discovery.py`、`runtime_ui/js/cards.py`

## 发布与回滚

- 发布方式：默认启用，仅增加解释字段和展示层，不改变发现 API 的主要调用方式。
- 回滚方式：移除 `match_reasons` 展示和评分中的说明拼装逻辑；排序结果本身仍可保留。
- 恢复路径：如果理由文案有噪音，可先收缩 UI 展示范围，再调整后端理由生成顺序。
