# 0008 - 忽略原因 + 反馈信号

## 任务元信息

- 任务 ID：`GS-P0-008`
- 优先级：`P0`
- 当前状态：`[x]`
- Sprint 排名：`8`
- 推荐 commit message：`feat: 添加忽略原因与反馈信号`

## 摘要

- 问题：当前忽略行为缺少结构化原因，发现系统也无法利用用户反馈改进结果质量。
- 为什么现在做：这是低风险增强项，能让发现系统开始形成本地偏好记忆。
- 结果：用户可以给出忽略原因，系统保存本地反馈信号。

## 战略映射

- 战略：把用户偏好保留在本地状态里，先做最小反馈闭环，不做云端训练或大规模排序重写。
- 路线图：对应 `docs/roadmap/ROADMAP.md` 的 P0 本地偏好记忆能力。
- Sprint：对应 `docs/sprints/CURRENT_TOP10.md` 排名 `8`。

## 当前状态

- 忽略动作会通过 prompt 采集原因，并在 `/api/state`、`/api/state/batch` 中透传 `feedback_reason`。
- 用户状态新增 `feedback_signals`，记录原因、次数、更新时间和状态。
- 发现评分会读取本地反馈信号，对命中过去被忽略原因的仓库做降权。
- 推荐理由中会展示“你曾因……忽略过它”，让用户能理解排序变化来源。

## 目标

- 让忽略动作可附带结构化原因。
- 保存本地反馈信号。
- 为未来的发现质量优化打基础。

## 非目标

- 不做复杂机器学习模型。
- 不上传用户反馈到云端。
- 不重写发现逻辑主干。

## 范围

### 范围内

- 忽略原因选择
- 本地反馈信号存储
- 后续排序可读取最小反馈

### 范围外

- 云端训练
- 高复杂个性化系统
- 大规模排序改写

## 架构触点

- 状态：`runtime/state.py`、`runtime/state_schema.py`
- HTTP：`runtime/http.py`
- 发现：`runtime_github/discovery.py`
- UI：`runtime_ui/js/actions.py`、`runtime_ui/js/overlays.py`

## 发布与回滚

- 发布方式：默认启用，全部保存在本地用户状态，不上传任何反馈数据。
- 回滚方式：停止采集 `feedback_reason`，移除 `feedback_signals` 读取 / 写入和发现降权逻辑。
- 恢复路径：如果发现反馈噪音过大，可保留原因采集但临时关闭排序消费逻辑。

  - 反馈信号可持久化
  - 不影响旧状态数据兼容
